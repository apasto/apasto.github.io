Running Ubiquiti AirView in 2026

April 15, 2026

I happen to own more than one Ubiquiti Airgrid M5, a pair of which I am about to use in a point-to-point link - in 2026. The power budget is a factor on the far end of this link, and I am quite content with the 1.9-2.3W peak I have measured under stress. Cost has been a factor too, I bought these used for cheap. Lesson learned: factor in the effort involved in de-rusting and repainting their grid reflectors.

Then there is the ’time tax’, since the side project these are meant to be used in has been 2.5 years in the making. But let’s not digress…

Anyway, these can run the nice AirView spectrum analysis tool. Or at least they could, until I tried again, recently.

The way AirView is launched is by downloading a Java Web Start jnlp file (airview.jnlp) from the AirOS Web UI. This, launched locally, pulls a jar from the device - not directly one, as we’ll see in a moment. When trying to do this from my PC (using IcedTea 1.8.8 and openjdk 21.0.10, on Ubuntu 24.04), I got this back:

net.sourceforge.jnlp.LaunchException: Fatal: Read Error: Could not read or parse the JNLP file.

Trying the same from a Windows machine with JRE resulted in the same error.

Looking at the (quite short) XML in airview.jnlp, there is a codebase attribute pointing to https://${DeviceIP}:443/${AirOsVersion}, then in a resources tag: <jar href="airview.jar" main="true" download="eager" />. However, trying to get e.g. this https://1.2.3.4:443/251204.1815/airview.jar returns a 404 (also note the non-existent version subdirectory, the meaning of which I am unsure). On a shell on the device, calling find / -name "*airview*", finds /usr/www/airview.jar.pack.gz - a gzipped and Pack200 compressed jar.

First thing first, we can wget --no-check-certificate https://1.2.3.4/airview.jar.pack.gz. Then calling gunzip airview.jar.pack.gz is not an issue. Now, it seems that Pack200 might be the culprit, given it was removed in Java 14: unpack200.

Docker helps as a time machine: we can call unpack from an old JDK version. openjdk is showing a deprecation warning for legacy JDK builds, so I switched to the eclipse-temurin images.

So, in the same dir where we gunzipped airview.jar.pack.gz to airview.jar.pack:

docker run --rm -v $(pwd):/data \
  -w /data eclipse-temurin:8 \
  unpack200 airview.jar.pack airview.jar

This should produce an airview.jar:

$ file airview.jar
airview.jar: Java archive data (JAR)

Which can be launched with:

java -cp airview.jar com.ubnt.app.AirViewer ubnt://1.2.3.4:18888

Now this will likely result in an error: the device will be found not to be in spectrum analysis mode (nothing listening on port 18888). Launching AirGrid the usual way (from the AirOS web UI, downloading the jnlp) worked out, after a bit of retries.
